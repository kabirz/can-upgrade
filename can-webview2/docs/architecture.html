前后端交互机制

WebView2 提供了一套基于 JSON 消息的双向通信机制。前端通过 chrome.webview API 发送消息，C++ 通过 PostWebMessageAsJson 推送消息。

通信管线

方向JS APIC++ API数据格式
JS → C++ chrome.webview.postMessage(obj) add_WebMessageReceived() JSON 对象
C++ → JS chrome.webview.addEventListener('message', cb) PostWebMessageAsJson(json) JSON 字符串
⚠️
注意：postMessage() 会自动将传入的 JS 对象序列化为 JSON。因此 JS 侧应直接传对象，不要再调用 JSON.stringify()，否则会导致双重编码，C++ 端无法解析。

JS → C++ 交互流程

1

用户点击按钮

例如用户点击"连接"按钮，触发 toggleConnect() 函数

2

JS 构造消息并发送

// JS 侧 chrome.webview.postMessage({ action: "connect", channelIndex: 0, baudIndex: 5 });
3

C++ 接收并解析

// C++ 侧 OnWebMessageReceived 回调 args->get_WebMessageAsJson(&message); // message = {"action":"connect","channelIndex":0,"baudIndex":5} auto action = GetJsonString(message, L"action"); // "connect"
4

调用 CanManager 执行业务

bool ok = g_app.canMgr->Connect( g_app.channels[channelIndex], BAUD_RATES[baudIndex] );
5

返回结果给 JS

PostJson(L"{\"event\":\"connectResult\",\"success\":true}");

C++ → JS 交互流程

C++ 主动推送消息的场景包括：日志输出、进度更新、操作完成通知。

1

业务层触发回调

CanManager 内部调用 m_logCb("CAN 连接成功") 或 m_progressCb(50)

2

通过 Windows 消息转发到 UI 线程

子线程不能直接操作 WebView2，使用 PostMessage(hWnd, WM_LOG_MESSAGE, ...) 回到 UI 线程

3

WndProc 构造 JSON 并推送

// WndProc 中处理 WM_UPDATE_PROGRESS wchar_t json[64]; swprintf(json, 64, L"{\"event\":\"progress\",\"percent\":%d}", percent); PostJson(json); // → webView->PostWebMessageAsJson(json)
4

JS 接收并更新 UI

// JS 侧 chrome.webview.addEventListener('message', (event) => { const data = event.data; // 已自动解析为对象 switch (data.event) { case 'progress': progressBar.value = data.percent; break; case 'log': logArea.value += data.message + '\n'; break; } });

线程安全

固件升级是耗时操作（可能数十秒），必须在子线程执行。但 WebView2 的 PostWebMessageAsJson 只能在 UI 线程调用。

线程模型

线程职责可调用 API
UI 线程
(主消息循环)		 WndProc、WebView2 操作、JSON 推送 PostWebMessageAsJson、WndProc
工作线程
(FlashThread)		 固件升级、CAN 通信 CanManager 方法、PostMessage

跨线程通信链路

FlashThread (工作线程) │ CanManager::FirmwareUpgrade() │ └─ m_progressCb(50) │ └─ OnProgress(50) │ └─ PostMessage(hWnd, WM_UPDATE_PROGRESS, 50, 0) ← 线程安全 ▼ WndProc (UI 线程) │ case WM_UPDATE_PROGRESS: │ └─ PostJson(L"{\"event\":\"progress\",\"percent\":50}") │ └─ webView->PostWebMessageAsJson(...) ← UI 线程安全 ▼ WebView2 → JS │ event.data = {event: "progress", percent: 50} ▼ DOM 更新 (progressBar.value = 50)
✓
关键设计：工作线程通过 PostMessage（Windows 消息队列）与 UI 线程通信，这是 Win32 中标准的线程安全通信方式。所有 WebView2 调用都在 UI 线程执行，不存在并发问题。

WebView2 vs 纯 Win32 API 对比

原版使用 Win32 Dialog API（DialogBox + 资源文件 .rc），新版使用 WebView2 加载 HTML 前端。以下是详细对比。

● WebView2 优势

  • HTML/CSS 布局比 Win32 Dialog 资源文件灵活得多
  • 改样式不需要重新编译，只改 HTML 文件
  • CSS 动画、渐变、圆角、阴影轻松实现
  • Flexbox 自动适配窗口大小，无需手动 WM_SIZE
  • UI 和逻辑完全分离，可维护性高
  • 自定义模态对话框比 MessageBox 美观
  • 同一套前端代码可移植到其他 WebView 方案
  • 开发者工具 (F12) 方便调试

● WebView2 劣势

  • 需要系统安装 WebView2 Runtime (~150MB)
  • 启动初始化约 1-3 秒，Win32 对话框几乎瞬间
  • 运行时内存占用高出约 20-50 倍
  • CMake 需下载 WebView2 SDK (NuGet)
  • 前后端交互需要理解消息传递机制
  • 不适合资源受限的嵌入式/老设备场景
  • 调试涉及 JS/C++ 两套工具链
